# ValueCell 学习路径：从新手到专家

## 学习目标

本文提供一条**基于当前仓库结构的真实可执行学习路径**。这里的"专家"指：能够独立解释系统架构、定位核心模块、完成定制扩展，并对关键设计做出负责任修改。

阅读本文后，你将能够：

- 评估自己当前所处的学习阶段
- 知道每个阶段该做什么、读什么、验证什么
- 通过自测检查清单确认是否达到阶段目标

---

## Stage 1：新手 —— 先把系统跑起来

### 目标

- 知道 ValueCell 是什么、能做什么
- 能在本地成功启动系统
- 完成一次最简单的 Agent 对话

### 学习内容

1. 阅读 [项目概览](./overview.md)——建立全局印象
2. 阅读 [快速上手](./getting-started.md)——按步骤操作
3. 执行 `bash start.sh` 启动系统
4. 在 Web UI 中选择一个 Agent，发起一次对话

### 关键认知

- ValueCell 是多智能体平台，不是单一聊天机器人
- 运行时配置在**系统目录**的 `.env` 中，不在仓库根目录
- 流式响应是系统的主干能力，不只是"打字机效果"

### 易错点提醒

- **不要混淆仓库根目录的 `.env.example` 和系统目录的 `.env`**：仓库中的 `.env.example` 是模板文件，包含所有可配置项的示例值和注释说明；系统目录中的 `.env` 才是运行时实际读取的配置文件。如果你修改了 `.env.example` 而没有创建系统目录的 `.env`，配置不会生效。
- **不要在启动失败时直接跳到源码排查**：绝大多数启动失败是因为环境变量缺失或端口冲突。先检查终端报错信息的最后几行，通常是明确的配置错误提示。

### 自测检查清单

- [ ] 我能用一句话解释 ValueCell 是什么
- [ ] 我知道 `.env` 文件在系统目录而不是仓库根目录
- [ ] 我成功启动了系统并能在浏览器中访问 `http://localhost:1420`
- [ ] 我与至少一个 Agent 完成了一次对话
- [ ] 我知道如何查看后端启动日志

### 预计耗时

约 **1-2 小时**（假设环境没有特殊阻碍）。

### 阶段衔接指南：Stage 1 -> Stage 2

在进入 Stage 2 之前，你应该能够独立启动系统、在浏览器中完成一次 Agent 对话，并且知道日志在哪里查看。如果你在启动环节卡住超过 30 分钟仍无法解决，说明环境配置可能存在特殊问题，建议先查阅 [getting-started](./getting-started.md) 中的常见问题部分。准备好后，下一步的核心是从"能跑起来"过渡到"能按需配置"。

## Stage 2：使用者 —— 会配置，知道边界

### 目标

- 能独立完成模型提供商配置
- 理解系统目录结构和数据存储位置
- 能区分"已实现"和"规划中"的功能

### 学习内容

1. 阅读 [配置说明](./configuration.md)——理解三层配置系统
2. 在 `.env` 中配置至少一个 LLM 提供商的 API Key
3. 了解系统目录中的 SQLite、LanceDB、知识库各自的作用
4. 查看 README 中的交易所支持状态，理解 ✅ 和 🟡 的区别

### 关键认知

- 配置优先级：环境变量 > `.env` > YAML 默认值
- Agent Card 和 Agent YAML 是两份不同的文件，缺一不可
- 路线图中的功能不等于已实现的功能

### 实践练习

1. **切换提供商**：在 `.env` 中切换主提供商（如从 OpenRouter 切换到 SiliconFlow），重启并观察 Agent 行为差异
2. **清理重建**：删除系统目录中的 `valuecell.db`，重启系统，观察数据库自动重建过程

### 易错点提醒

- **不要把 README 中描述的所有功能都当成已实现的**：README 和路线图中包含部分规划中的功能。判断功能是否已实现的最可靠方式是查看对应目录下是否有实际代码，而不是仅凭文档描述。
- **不要忽略 Agent Card 和 Agent YAML 的区别**：Agent Card（JSON 格式）是服务发现和能力声明，Agent YAML 是运行参数配置。两者的字段名称和用途不同，混用会导致 Agent 无法被正确识别。

### 自测检查清单

- [ ] 我知道三层配置优先级及其设计意图
- [ ] 我成功配置了至少一个模型提供商
- [ ] 我知道 Agent Card 和 Agent YAML 的区别
- [ ] 我能区分哪些交易所是"已测试"、哪些是"部分测试"
- [ ] 我知道数据库文件在哪里，以及如何安全清理重建

### 预计耗时

约 **2-3 小时**。

### 阶段衔接指南：Stage 2 -> Stage 3

进入 Stage 3 之前，你应该能独立切换模型提供商并理解配置从哪里来、数据库在哪里。如果你无法说出三层配置的优先级关系，建议回到 [configuration](./configuration.md) 重新阅读。准备好后，下一步是从"配置使用者"过渡到"代码阅读者"，开始接触前端和后端的入口文件。

## Stage 3：开发者入门 —— 读懂主要入口

### 目标

- 能找到前端和后端的关键入口文件
- 理解一次请求从发起到响应的大致流向
- 能运行测试和代码检查

### 学习内容

1. 阅读 [系统架构详解](./architecture.md)——理解链路
2. 阅读 [流式事件系统深度解析](./streaming-deep-dive.md)——理解事件分类与处理管线
3. 阅读 [源码导读](./source-guide.md)——找到入口
4. 通读以下核心文件（不要求全部理解细节）：

**前端：**
- `frontend/src/root.tsx` — 根布局
- `frontend/src/routes.ts` — 路由定义
- `frontend/src/lib/sse-client.ts` — SSE 客户端

**后端：**
- `python/valuecell/server/api/app.py` — 应用工厂
- `python/valuecell/core/coordinate/orchestrator.py` — 编排器
- `python/valuecell/core/types.py` — 核心类型定义

5. 运行测试：

```bash
# Python 测试
cd python && uv run pytest

# 前端类型检查
cd frontend && bun run typecheck
```

### 关键认知

- 前端通过自定义 SSE 客户端消费流式事件（不是原生 EventSource）
- 后端编排器使用后台 producer 模式，前端断开后任务仍可继续
- Agent 是独立服务化节点，通过 A2A 协议通信

### 实践练习

1. **追踪一条请求**：在前端发起一条消息，同时在后端终端观察日志输出，尝试对应"请求接入 → Super Agent 分流 → 规划/直通 → 执行 → 事件返回"这几个阶段
2. **阅读一个完整 Agent**：从头到尾阅读 `python/valuecell/agents/news_agent/` 的所有文件

### 易错点提醒

- **不要跳过 `core/types.py`**：这个文件定义了整个系统中所有核心数据类型和事件类型。跳过它直接读其他模块的源码，你会频繁遇到不认识的类型引用，导致阅读效率极低。建议把 `types.py` 作为源码阅读的第一个文件，逐个理解其中定义的类型。
- **不要把前端 SSE 客户端等同于浏览器原生 `EventSource`**：ValueCell 的 SSE 客户端基于 `fetch + ReadableStream` 实现，支持更灵活的事件解析和错误处理。原生 `EventSource` 不支持自定义请求头和 POST 方法，无法满足系统的认证需求。

### 自测检查清单

- [ ] 我能说出前端根布局注入了哪些全局 Provider
- [ ] 我知道 SSE 客户端基于 `fetch + ReadableStream` 而非原生 `EventSource`
- [ ] 我能指出后端编排器的主入口函数名
- [ ] 我知道 `core/types.py` 中定义了哪些事件类型
- [ ] 我成功运行了 Python 测试和前端类型检查

### 预计耗时

约 **4-6 小时**（含源码阅读时间）。

### 阶段衔接指南：Stage 3 -> Stage 4

进入 Stage 4 之前，你应该能说出请求从发起到响应经过了哪些关键节点，并且至少完整阅读过一个 Agent 的源码。如果你无法描述请求的大致流转路径，建议回到 [architecture](./architecture.md) 重新建立全局认知。准备好后，下一步是从"能读懂入口"过渡到"能定位问题"，重点是理解事件层的内部机制。

## Stage 4：进阶开发 —— 能定位问题，能做小改动

### 目标

- 能区分前端问题、后端问题、Agent 问题、配置问题
- 能修改已有模块而不破坏主链路
- 理解流式事件的完整生命周期

### 学习内容

1. 精读 `core/event/` 目录下的三个关键文件：
   - `router.py` — 事件路由（将底层事件转为前端可用的响应）
   - `buffer.py` — 事件缓冲（分配稳定 `item_id`，聚合分片消息）
   - `service.py` — 统一门面（创建响应、缓冲、持久化三合一）
2. 阅读 [核心设计原理深度解析](./principles.md)——理解架构决策背后的动机和权衡
3. 阅读 [前端架构深度解析](./frontend-architecture.md)——理解前端状态管理和 SSE 通信
4. 理解 `ResponseBuffer` 如何实现"消息逐步生长"的效果
5. 了解 `planner passthrough` 机制及其适用场景

### 关键认知

- 流式消息不是随便拼字符串，而是依赖稳定的 `item_id` 和事件契约
- 前端 8 个渲染器对应后端不同的事件类型
- 不是所有请求都会进入完整的 Planner 推理——有些 Agent 使用直通模式

### 实践练习

1. **修改前端渲染**：在某个 Renderer 中添加一行 `console.log`，观察事件如何流入渲染器（例如编辑 `frontend/src/components/valuecell/renderer/markdown-renderer.tsx`，在组件函数体开头添加 `console.log('rendering markdown:', content)`）
2. **修改后端日志**：在 `orchestrator.py` 的关键分支点添加日志，观察请求的分流路径
3. **调试一次完整请求**：开启 `AGENT_DEBUG_MODE=true`，发一条消息，阅读完整调试日志

### 易错点提醒

- **不要混淆 `TaskStatusEvent`（外部事件）和 `TaskStatus`（内部状态）**：`TaskStatusEvent` 是发送给前端的流式事件，属于事件协议的一部分；`TaskStatus` 是后端内部的任务状态枚举，用于编排器内部逻辑。两者的值和含义不完全相同，混用会导致对事件流的理解出现偏差。
- **不要假设所有请求都会走完整的 Planner 推理链路**：部分 Agent 使用直通模式（passthrough），跳过 Planner 直接进入执行。如果你的调试目标是理解 Planner 行为，需要先确认当前请求是否真的经过了 Planner。

### 自测检查清单

- [ ] 我能解释 `ResponseRouter`、`ResponseBuffer`、`EventResponseService` 三者的职责
- [ ] 我知道为什么前端能看到"一条消息逐渐长出来"的效果
- [ ] 我能说出至少两种执行模式（标准规划 vs 直通）的区别
- [ ] 我修改了一个模块，系统仍然正常运行

### 预计耗时

约 **6-10 小时**。

### 阶段衔接指南：Stage 4 -> Stage 5

进入 Stage 5 之前，你应该能独立修改一个模块且不破坏主链路，并且理解流式事件从产生到渲染的完整路径。如果你无法解释 `ResponseRouter`、`ResponseBuffer`、`EventResponseService` 三者的职责分工，建议回到 event 层的源码重新阅读。准备好后，下一步是从"读懂和微调"过渡到"从零构建"，开始创建自己的 Agent。

## Stage 5：扩展 —— 能新增 Agent

### 目标

- 能从零新增一个可工作的 Agent
- 正确配置 Agent Card 和 Agent YAML
- 让前端看到该 Agent 的流式输出

### 学习内容

1. 阅读 [Agent 开发与扩展](./agent-extension.md)——完整接入路径
2. 阅读 [数据适配器深度解析](./data-adapter.md)——理解市场数据获取机制
3. 对照 `ResearchAgent` 或 `NewsAgent` 理解现有实现
4. 实现一个最小 Agent 并完成闭环验证

### 关键认知

- Agent 的三个组成部分：代码实现、Agent Card、Agent YAML
- `stream()` 方法必须是异步的，必须以 `streaming.done()` 结束
- Agent Card 中的 `name` 必须与类名完全一致

### 实践练习

**创建一个 HelloAgent**，完成以下闭环：

1. 创建目录和文件（`__init__.py`、`__main__.py`、`core.py`）
2. 实现最简单的 `stream()` 方法
3. 创建 Agent Card（JSON）
4. 创建 Agent YAML 配置
5. 单独启动并验证
6. 接入后端并在前端看到输出

### 易错点提醒

- **不要忘记 `create_wrapped_agent()` 中 Agent Card 的 `name` 必须与类名完全一致**：这是 Agent 被系统正确发现和注册的前提条件。如果 `name` 字段与 Python 类名不一致（哪怕是大小写差异），Agent 将无法被路由到，前端会显示找不到 Agent 的错误。
- **不要在 `stream()` 方法中忘记调用 `streaming.done()`**：这会导致前端永远无法收到响应结束信号，表现为消息一直处于"生成中"状态。务必确保所有代码路径（包括异常路径）最终都会调用 `streaming.done()`。

### 自测检查清单

- [ ] 我的新 Agent 能在系统中被发现
- [ ] 我能在前端与它对话并收到流式响应
- [ ] 我理解了 `streaming.message_chunk()` 和 `streaming.done()` 的作用
- [ ] 我知道如何给 Agent 添加工具（Tool）

### 预计耗时

约 **4-8 小时**（首次接入）。

### 阶段衔接指南：Stage 5 -> Stage 6

进入 Stage 6 之前，你应该已经从零创建过一个可工作的 Agent，并且理解了 Agent 的三个组成部分（代码、Card、YAML）如何协同工作。如果你创建的 Agent 还无法在前端正常对话，不要急于进入下一阶段，先回到 [agent-extension](./agent-extension.md) 对照排查。准备好后，下一步是从"能扩展"过渡到"能解释设计并安全改动核心"。

### 实践建议

## 学习策略建议

在按照上述路线学习的过程中，以下策略可以帮助你提升效率、减少挫折感。

### 管理认知负荷

不要试图一次性理解所有细节。正确的阅读顺序是：先读概览文档建立全局印象，再读架构文档理解模块关系，最后才深入源码。如果你发现自己读源码时频繁"看不懂这一行在做什么"，说明你还没有建立足够的上下文，应该退回到上层文档重新理解整体设计。每个阶段都有明确的学习边界，完成当前阶段的检查清单后再进入下一阶段，避免跨阶段跳跃式学习。

### 有效记笔记

阅读文档和源码时，重点记录三类信息：

- **入口点**：每个模块的入口函数或入口文件是什么？例如前端入口是 `root.tsx`，后端 API 入口是 `app.py`，编排入口是 `orchestrator.py`。掌握了入口点，就掌握了阅读路径。
- **契约**：模块之间的接口约定是什么？例如 `stream()` 方法必须以 `streaming.done()` 结束，Agent Card 的 `name` 必须与类名一致。契约是系统稳定运行的基石。
- **边界**：每个模块负责什么、不负责什么？例如 `ResponseRouter` 负责事件路由但不负责缓冲，`ResponseBuffer` 负责缓冲但不负责持久化。理解边界才能准确定位问题。

建议使用 Markdown 文件记录，每个阶段结束时整理一次。不要记录代码细节，而是记录"为什么这样设计"。

### 验证理解的方法

理解程度的最佳检验方式是**用自己的话重新解释**。每完成一个阶段的学习，尝试以下方式验证：

- **费曼法**：假装向一个不了解项目的朋友解释系统如何工作。如果你在某个地方卡住了，说明那个地方你还没有真正理解。
- **写总结**：用 200 字以内描述当前阶段学到的核心内容。如果你写不出来，说明你的理解还不够结构化。
- **预测行为**：在做某个操作之前，先预测系统会怎样响应，然后实际操作验证。预测正确说明理解到位，预测错误则是极好的学习机会。

### 用调试作为学习工具

调试不只是修 bug 的手段，更是理解系统运行机制的利器。建议在学习过程中善用以下调试手段：

- 开启 `AGENT_DEBUG_MODE=true` 环境变量，系统会输出完整的请求处理日志，包括事件流转、Agent 调用链、状态变更等。这是理解系统行为的最高效方式。
- 在关键函数入口添加 `print()` 或 `logger.debug()`，观察函数何时被调用、参数是什么。例如在 `orchestrator.py` 的 `run()` 方法中添加日志，可以直观看到请求的完整生命周期。
- 在前端渲染器中添加 `console.log()`，观察事件如何流入组件并被渲染。这对于理解前后端事件协议特别有效。

调试日志是"活文档"——它展示的是系统实际的行为，而不是文档中描述的理想行为。当你发现两者不一致时，恭喜你发现了知识的盲区。

## 学习时要避免的三个误区

### 误区 1：把 README 当成全部事实来源

README 适合建立整体印象，但关键实现细节必须回到源码确认。中文文档以当前代码为准。

### 误区 2：上来就钻细节

如果没有先建立全局图谱（读 overview 和 architecture），直接看源码会非常碎。先见森林，再见树木。

### 误区 3：把"会运行"误当成"已理解"

真正理解项目意味着你知道：

- 请求怎么流转（前端 → API → Orchestrator → Planner → Agent → 事件 → 前端）
- 配置从哪来（三层优先级、系统目录）
- 事件如何被渲染和持久化（Router → Buffer → Service → Store）
- 新 Agent 怎样被系统识别（Card + YAML + 代码三者缺一不可）
