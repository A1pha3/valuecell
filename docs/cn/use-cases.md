# ValueCell 使用场景与实战指南

## 学习目标

阅读本文后，你将能够：

- 列举 ValueCell 当前已实现的五大使用场景及其对应的 Agent
- 按步骤完成金融研究、新闻追踪、交易策略等实际操作
- 解释多 Agent 协作的工作方式，包括编排器如何自动调度
- 通过在线版本和桌面应用两种方式使用 ValueCell
- 严格区分"已实现"、"部分测试"和"规划中"的功能边界
- 识别交易策略场景中的关键风险点并采取相应的安全措施

---

> **风险提示**：ValueCell 的交易策略功能涉及实盘操作。加密货币市场波动剧烈，可能导致资金损失。在使用实盘交易前，**务必先在模拟盘充分验证策略**。文档中对交易所支持状态的标记（✅ 已测试 / 🟡 部分测试）反映的是技术对接完成度，不代表对交易安全的任何保证。

---

## 1. 使用场景总览

ValueCell 当前已实现的使用场景可分为以下几类：

| 场景 | 核心能力 | 涉及的 Agent | 状态 |
|------|----------|-------------|------|
| **金融研究** | SEC 财报分析、13F 持仓解析、知识库综合 | ResearchAgent | ✅ 已实现 |
| **新闻追踪** | 实时新闻搜索、定时推送、突发新闻监控 | NewsAgent | ✅ 已实现 |
| **交易策略** | 自然语言策略生成、网格交易、模拟盘验证 | PromptBasedStrategyAgent、GridStrategyAgent | ✅ 已实现（交易所支持程度不同） |
| **多 Agent 协作** | 编排器自动调度多个 Agent 协同工作 | Orchestrator + 全部 Agent | ✅ 已实现 |
| **开发者扩展** | 新增 Agent、数据源、模型提供商 | 开发者自行实现 | ✅ 框架已就绪 |

> **注意**：以下场景为规划中，当前尚未实现——欧股研究、商品期货分析、外汇交易、期权策略。这些内容出现在项目的 README 路线图中，不代表当前可用。

## 2. 金融研究场景

### 2.1 场景说明

金融研究场景由 **ResearchAgent**（端口 10004）提供支持。它能够分析 SEC（美国证券交易委员会）公开财报（10-K、10-Q、13F、8-K），结合内部知识库生成可溯源的研究摘要和数据提取。

**为什么用 ResearchAgent 而不是直接让 LLM 分析？** ResearchAgent 封装了专业的财报解析工具，能够结构化提取财务指标（营收、EPS、现金流等），而通用 LLM 对财报的原始理解往往不够精确。

### 2.2 前提条件

| 条件 | 说明 |
|------|------|
| SEC_EMAIL 环境变量 | ResearchAgent 访问 SEC EDGAR 数据需要此配置（用于 User-Agent 标识） |
| 模型提供商 API Key | 至少配置一个 LLM 提供商 |
| 系统正常运行 | 后端和前端均已启动 |

### 2.3 配置步骤

**第一步：配置 SEC_EMAIL**

在系统目录的 `.env` 文件中添加（macOS 路径为 `~/Library/Application Support/ValueCell/.env`）：

```bash
# 使用你自己的邮箱（SEC 要求提供联系方式作为 User-Agent 标识）
SEC_EMAIL=your_email@example.com
```

**第二步：确保模型提供商已配置**

ResearchAgent 需要调用 LLM 来分析财报内容。至少配置一个模型提供商的 API Key。推荐使用 OpenRouter（模型最全）或 SiliconFlow（国内友好）。详见 [getting-started.md](./getting-started.md) 第 4 节。

**第三步：重启后端服务**

修改 `.env` 后需要重启后端才能生效：

```bash
# 停止当前运行的服务，然后重新启动
bash start.sh
```

### 2.4 操作示例

**操作一：提取财务数据**

在 Web UI 中选择 ResearchAgent，输入：

```text
提取苹果公司（AAPL）最新 10-K 财报的营收、净收入、EPS 和现金流数据
```

你将看到以下流式输出过程：

1. **推理过程**（`reasoning` 事件）：Agent 的思考链——"需要先搜索 AAPL 的 10-K 报告..."
2. **工具调用**（`tool_call_started` / `tool_call_completed`）：显示正在查询 SEC EDGAR
3. **文本内容**（`message_chunk`）：逐步展示提取的财务指标
4. **结构化摘要**：最终汇总为可读的分析报告

**操作二：分析 13F 持仓**

输入：

```text
分析伯克希尔哈撒韦最新的 13F 持仓报告，列出前十大持仓和仓位变动
```

ResearchAgent 会调用 `analyze_13f_filings` 技能，解析机构投资者的持仓变化。13F 报告由管理资产超过 1 亿美元的机构按季度向 SEC 提交，是追踪机构投资者动向的重要数据源。

**操作三：综合知识库搜索**

输入：

```text
结合知识库中关于英伟达（NVDA）的信息，分析其 AI 业务增长趋势
```

ResearchAgent 会调用 `synthesize_context` 技能，同时检索 SEC 文件和内部知识库，综合多个信息源提供分析。

**操作四：文件监控与通知**

输入：

```text
帮我监控特斯拉（TSLA）的 SEC 文件提交，有新文件时通知我
```

ResearchAgent 会设置监控任务，当有新的 8-K、10-K 等文件提交时，通过推送通知告知你。前端会渲染为 `filtered_card_push_notification` 类型的通知卡片。

**操作五：A 股在线研究（通过在线版本）**

访问 [valuecell.ai](https://valuecell.ai)，输入：

```text
分析贵州茅台（600519）最近的财务表现和行业地位
```

注意：A 股研究当前通过在线版本 valuecell.ai 使用，本地部署版本暂不支持 A 股研究功能。

### 2.5 ResearchAgent 能力总结

| 技能（Skill） | 说明 | 触发示例 |
|---------------|------|----------|
| `extract_financials` | 从 10-K、10-Q 等财报中提取财务指标 | "提取 AAPL 最新财报数据" |
| `analyze_13f_filings` | 解析机构 13F 持仓报告和仓位变动 | "分析伯克希尔的 13F 持仓" |
| `synthesize_context` | 结合知识库搜索结果提供综合解读 | "结合知识库分析 NVDA" |
| `monitor_filings_and_notify` | 跟踪公司文件提交并发送摘要通知 | "监控 TSLA 的 SEC 文件" |

## 3. 新闻追踪场景

### 3.1 场景说明

新闻追踪场景由 **NewsAgent**（端口 10005）提供支持。它提供实时新闻检索与分析能力，支持个性化定时推送和突发新闻监控。

### 3.2 操作示例

**操作一：实时新闻搜索**

在 Web UI 中选择 NewsAgent，输入：

```text
今天有哪些重要的金融市场新闻？
```

NewsAgent 的处理流程：

1. 通过网络搜索工具获取当前新闻
2. 流式展示搜索过程中的工具调用状态
3. 返回新闻条目的标题、摘要和来源
4. 提供 AI 对新闻的综合分析

**操作二：设置定时新闻推送**

输入：

```text
每天早上 9 点给我推送美股市场的最新动态
```

NewsAgent 支持设置定时推送任务。设置后：

- 前端渲染 `scheduled_task_controller` 组件，用于管理定时任务
- 你可以查看当前活跃的定时任务列表
- 可以启用、暂停或查看历史执行结果
- 每次执行时，系统通过 `scheduled_task_result` 组件展示结果

**操作三：突发新闻监控**

输入：

```text
监控关于美联储利率决议的突发新闻
```

NewsAgent 会持续监控相关主题。当有新的突发新闻时，通过 `notify()` 方法主动推送通知（`NotifyResponseEvent`），前端渲染为通知卡片。

### 3.3 NewsAgent 能力总结

| 能力 | 说明 | 实现方式 |
|------|------|----------|
| 实时新闻搜索 | 通过网络搜索获取当前新闻 | `stream()` 方法 + 工具调用 |
| 突发新闻监控 | 监控并获取紧急新闻更新 | `stream()` 方法 + 搜索工具 |
| 金融市场新闻 | 综合性金融市场和商业新闻 | `stream()` 方法 + 综合搜索 |
| 定时推送 | 按设定的时间周期性推送新闻摘要 | 定时任务 + `notify()` 方法 |

## 4. 交易策略场景

### 4.1 场景说明

交易策略场景由两个 Agent 共同支撑：

| Agent | 端口 | 执行模式 | 说明 |
|-------|------|----------|------|
| **PromptBasedStrategyAgent** | 10006 | `planner_passthrough`（直通） | 根据自然语言描述生成交易策略 |
| **GridStrategyAgent** | 10007 | `planner_passthrough`（直通） | 网格交易策略 |

**直通模式（Planner Passthrough）的含义**：这两个 Agent 跳过完整的 Planner 推理流程，用户请求直接交给 Agent 执行。这适用于交互模式简单、不需要复杂规划的场景。

两个策略 Agent 默认在前端隐藏（Agent Card 中 `metadata.hidden: true`），由系统编排器自动调度，不需要用户手动选择。

> **重要安全提醒**
>
> 交易策略涉及实盘操作，存在资金风险。请严格遵守以下安全原则：
>
> 1. **先模拟盘**：任何策略必须先在模拟盘（Paper Trading）充分验证
> 2. **理解逻辑**：充分理解策略逻辑和风险参数后再使用
> 3. **小仓位开始**：先用小仓位试运行，确认无误后再逐步放大
> 4. **持续监控**：时刻关注市场变化和系统运行状态
> 5. **设置止损**：所有策略都应设置合理的止损参数

### 4.2 支持的市场与交易所

**行情数据来源：**

| 市场 | 数据来源 | 说明 |
|------|----------|------|
| 美股 | Yahoo Finance（`yfinance`） | 通过 `yfinance` 库获取实时和历史行情 |
| 港股 | Yahoo Finance（`yfinance`） | 通过 `yfinance` 库获取实时和历史行情 |
| A 股 | AKShare + BaoStock | 通过东方财富等国内数据源获取行情 |
| 加密货币 | 交易所 API | 通过交易所接口获取行情和执行交易 |

**加密货币交易所支持状态：**

| 交易所 | 状态 | 交易类型 | 配置要求 |
|--------|------|----------|----------|
| Binance | ✅ 已测试 | USDT 永续合约 | API Key + Secret |
| Hyperliquid | ✅ 已测试 | USDC 保证金 | 主钱包地址 + API 钱包私钥 |
| OKX | ✅ 已测试 | USDT 永续合约 | API Key + Secret + 密码 |
| Coinbase | 🟡 部分测试 | — | 代码已完成，未完全测试 |
| Gate.io | 🟡 部分测试 | — | 代码已完成，未完全测试 |
| MEXC | 🟡 部分测试 | — | 代码已完成，未完全测试 |

> **注意**：
> - 当前仅支持**合约交易**（Futures Trading，现货以 1 倍合约实现）
> - Binance 仅支持国际站，不支持国内站
> - 使用前务必确保合约账户余额充足
> - ✅ 标记表示经过基本功能测试，不代表对所有交易场景都已验证

### 4.3 操作步骤

**第一步：配置交易所 API**

在系统目录的 `.env` 文件中配置目标交易所的 API 信息。

以 OKX 为例（当前 `.env.example` 中已预定义 OKX 相关变量）：

```bash
# 模拟盘（推荐先用模拟盘）
OKX_NETWORK=paper
OKX_API_KEY=your_api_key
OKX_API_SECRET=your_api_secret
OKX_API_PASSPHRASE=your_passphrase
OKX_ALLOW_LIVE_TRADING=false    # 必须显式设为 true 才能使用实盘
```

> **注意**：Binance 和 Hyperliquid 的 API 凭据通过策略运行时参数传入，不在 `.env.example` 中预定义。交易所连接代码位于 `python/valuecell/agents/common/trading/execution/ccxt_trading.py`。

**第二步：使用自然语言策略**

通过系统编排器（不是直接与某个策略 Agent 对话）输入请求：

```text
在 Binance 上对 BTC/USDT 设置一个均值回归策略，当价格跌破 20 日均线 2% 时买入，涨破 2% 时卖出，每次仓位不超过总资金的 5%
```

系统处理流程：

1. Orchestrator 接收请求
2. Super Agent 判断这是一个策略类请求
3. Planner 将请求路由到 PromptBasedStrategyAgent
4. PromptBasedStrategyAgent 根据自然语言描述生成交易指令
5. 前端通过流式事件实时展示策略生成过程和工具调用

**第三步：使用网格策略**

输入：

```text
对 ETH/USDT 在 2000-3000 价格区间设置 10 格等差网格，每格投入 100 USDT
```

系统会将请求路由到 GridStrategyAgent，根据参数设置网格交易。

**第四步：模拟盘验证**

ValueCell 支持模拟盘交易，用于在不使用真实资金的情况下验证策略：

1. 在交易所配置中选择模拟盘模式（如 OKX 的 `OKX_NETWORK=paper`）
2. 执行策略，观察模拟环境中的表现
3. 分析关键指标：收益率、最大回撤、胜率、夏普比率
4. 确认策略逻辑符合预期后再考虑切换到实盘

> **注意**：ValueCell 当前不内置这些指标的计算能力。以上指标需要通过 Agent 的研究分析功能间接获取，或由用户根据策略执行结果自行计算。

**第五步：切换到实盘（谨慎操作）**

在模拟盘验证充分后：

1. 确认交易所 API 配置正确且账户余额充足
2. 将配置切换到实盘模式（如 `OKX_NETWORK=mainnet`，同时 `OKX_ALLOW_LIVE_TRADING=true`）
3. 建议先用最小仓位试运行
4. 持续监控运行状态和收益表现

### 4.4 HITL（人机确认）机制

策略类操作涉及资金风险，系统在关键节点会通过 **HITL（Human-in-the-Loop）** 机制请求用户确认：

- 当策略参数不完整时 → 系统暂停并要求补充信息
- 当操作存在明显风险时 → 系统要求用户明确确认
- 确认通过后 → 系统从断点继续执行

在前端，你会看到 `plan_require_user_input` 事件触发的确认界面，显示需要确认的内容和操作选项。

### 4.5 常见问题

**Q：为什么策略 Agent 在前端看不到？**

策略 Agent（PromptBasedStrategyAgent 和 GridStrategyAgent）的 Agent Card 中设置了 `metadata.hidden: true`，默认不在 Agent 列表中显示。你通过主对话界面发送策略类请求时，系统会自动将请求路由到对应的策略 Agent。

**Q：交易所 API 配置后策略不执行？**

排查步骤：
1. 确认 API Key 权限包含交易权限（不需要提币权限）
2. 确认 API Key 未过期
3. 确认合约账户余额充足（不是现货账户）
4. 检查后端日志中的错误信息

## 5. 多 Agent 协作场景

### 5.1 场景说明

ValueCell 的核心价值之一是多个 Agent 的协同工作。用户不需要手动选择 Agent——系统的编排层（Orchestrator，编排器）会根据请求内容自动调度。

**为什么需要编排器？** 如果每个 Agent 独立工作，用户需要自己判断"这个问题该问谁"。编排器通过 Super Agent 做初步分流，再由 Planner 决定调用哪些 Agent，实现了用户只管提问、系统自动分配的体验。

### 5.2 协作流程

当你输入一个复合型请求时，系统的处理流程：

```text
用户请求 → Orchestrator → Super Agent（分流判断）
                              ↓
                    ┌── 直接回答（简单 Q&A，如"什么是市盈率"）
                    └── Planner（复杂请求）
                              ↓
                    分析请求，选择合适的 Agent
                              ↓
                    ┌── ResearchAgent（研究类任务）
                    ├── NewsAgent（新闻类任务）
                    ├── PromptBasedStrategyAgent（策略类任务）
                    └── GridStrategyAgent（网格类任务）
                              ↓
                    Agent 执行 → 流式事件返回 → 前端渲染
```

### 5.3 操作示例

**示例一：研究与新闻结合**

输入：

```text
帮我分析特斯拉的最新财报表现，并结合最近的相关新闻给出综合评估
```

系统处理：Planner 自动分析请求并调度多个 Agent，用户不需要手动选择。ResearchAgent 分析财报数据 → NewsAgent 搜索相关新闻 → 综合呈现结果。前端通过 `subagent_conversation` 组件标记每个子 Agent 的会话边界，让你清楚地看到哪个 Agent 贡献了哪部分内容。

**示例二：研究到策略的延续**

输入：

```text
根据英伟达最近的财报和新闻走势，帮我制定一个网格交易策略
```

系统处理：Planner 自动分析请求并调度多个 Agent，用户不需要手动选择。ResearchAgent 完成基本面研究 → 将结论作为上下文传递给 GridStrategyAgent → 生成策略参数建议。

### 5.4 Agent 发现与选择机制

Planner 选择 Agent 的依据来自 Agent Card 中的声明：

| 依据 | 来源 | 说明 |
|------|------|------|
| Agent 能力 | `skills` 字段 | Planner 通过 skill 描述匹配用户意图 |
| Agent 说明 | `description` 字段 | 帮助 Planner 理解 Agent 的定位 |
| 执行模式 | `metadata.planner_passthrough` | 决定是否跳过完整规划 |

### 5.5 当前 Agent 能力对照

| Agent | 声明的 Skills | 推送通知 | 执行模式 | 前端可见 |
|-------|-------------|----------|----------|----------|
| ResearchAgent | extract_financials, analyze_13f_filings, synthesize_context, monitor_filings_and_notify | 不支持 | 标准规划 | 可见 |
| NewsAgent | 实时新闻搜索, 突发新闻监控, 金融市场新闻 | 支持 | 标准规划 | 可见 |
| PromptBasedStrategyAgent | strategy_run | 支持 | 直通模式 | 隐藏 |
| GridStrategyAgent | 无（由 Planner 直通处理） | 支持 | 直通模式 | 隐藏 |

## 6. 开发者扩展场景

### 6.1 场景说明

ValueCell 的架构设计鼓励社区开发者扩展系统能力。框架层面已就绪，你可以通过三种主要方式扩展：

1. **新增 Agent**：实现新的金融能力（如技术分析 Agent、风控 Agent）
2. **接入新数据源**：添加新的市场数据适配器
3. **接入新模型提供商**：添加新的 LLM 服务

### 6.2 新增 Agent 与接入新数据源

ValueCell 支持通过创建新的 Agent 扩展金融分析能力，也支持通过自定义数据适配器接入新的市场数据源。Agent 开发涉及 Python 实现、Agent Card 和 YAML 配置三个部分；数据适配器则需继承 `BaseDataAdapter` 并在系统中注册。

这两部分的开发流程、文件结构和配置细节在专门文档中有完整说明，请分别参阅 [Agent 开发与扩展](./agent-extension.md) 和 [数据适配器开发指南](./data-adapter.md)。

### 6.3 接入新模型提供商

ValueCell 当前支持 9 个 LLM 提供商，新增提供商只需：

1. 在 `python/configs/providers/` 下创建新的 YAML 配置文件
2. 参考现有的 `openai-compatible.yaml` 格式
3. 在 `.env` 中添加对应的 API Key
4. 系统会自动检测新的提供商配置

详见 [configuration.md](./configuration.md) 第 4 节。

## 8. 桌面应用使用

### 8.1 安装

从 [GitHub Releases](https://github.com/ValueCell-ai/valuecell/releases) 下载对应平台的安装包：

| 平台 | 格式 | 说明 |
|------|------|------|
| macOS | `.dmg` | 拖拽安装到 Applications |
| Windows | `.msi` 或 `.exe` | 运行安装程序 |

桌面应用基于 **Tauri v2** 打包，内含完整 Python 后端，无需用户单独安装 Python。

### 8.2 首次配置

安装后的首次配置流程：

1. 启动应用
2. 进入设置页面（`/setting/models`）
3. 配置至少一个模型提供商的 API Key（推荐 OpenRouter 或 SiliconFlow）
4. 返回首页开始使用

### 8.3 桌面应用的特性

| 特性 | 说明 |
|------|------|
| 自动更新 | 内置更新检查，每 60 分钟检查一次，有新版本时提示升级 |
| 后端健康检查 | 启动时自动检测后端服务状态，不可用时显示加载画面 |
| 数据本地化 | 所有敏感数据（API Key、数据库、知识库）保存在本地系统目录 |
| 单实例运行 | 通过 `tauri-plugin-single-instance` 防止多实例同时运行 |

数据存储位置：

| 操作系统 | 路径 |
|----------|------|
| macOS | `~/Library/Application Support/ValueCell/` |
| Linux | `~/.config/valuecell/` |
| Windows | `%APPDATA%\ValueCell\` |

### 8.4 桌面应用 vs 本地开发

| 特性 | 桌面应用 | 本地开发 |
|------|----------|----------|
| 安装方式 | 下载安装包 | `git clone` + `bash start.sh` |
| 更新方式 | 应用内自动更新 | `git pull` 手动更新 |
| 适用场景 | 日常使用 | 开发调试、功能扩展 |
| 前置要求 | 无（内含 Python 后端） | Python 3.12+、bun、uv |
| 自定义能力 | 有限 | 完全自定义 |

## 11. 最佳实践

以下是使用 ValueCell 各功能模块时的实用建议，帮助你获得更准确的结果和更安全的操作体验。

### 11.1 如何高效地向 Agent 提问

不同 Agent 对查询方式的响应各有特点。掌握以下技巧可以显著提升结果质量：

- **ResearchAgent**：尽量提供具体的股票代码（如 AAPL 而非"苹果"），明确指明需要的财报类型（10-K 年报、10-Q 季报、13F 持仓），并指定关注的财务指标（营收、EPS、现金流）。例如，"提取 AAPL 最新 10-K 的营收和 EPS"比"分析一下苹果"能得到更精确的结果。
- **NewsAgent**：使用明确的时间范围（如"今天""本周"）和具体的主题关键词（如"美联储利率决议"而非"经济新闻"），有助于缩小搜索范围并提高相关性。
- **策略 Agent**：通过编排器提交策略请求时，完整描述交易标的、策略逻辑、仓位比例和风控参数。缺少关键参数时系统会通过 HITL 机制要求补充，但一次性提供完整信息能减少交互轮次。

### 11.2 如何验证研究结果

ResearchAgent 提供的财报分析结果基于 SEC EDGAR 的公开数据，但仍建议进行以下验证：

1. **交叉核对原始文件**：ResearchAgent 在输出中会引用数据来源。对于关键财务数据，建议前往 SEC EDGAR（sec.gov/cgi-bin/browse-edgar）查看原始文件进行核实。
2. **注意报告周期**：确认数据对应的时间区间是年报（10-K）还是季报（10-Q），避免将不同周期的数据直接比较。
3. **关注脚注与附注**：SEC 财报中的脚注通常包含重要的会计政策变更和一次性项目说明，这些可能影响数据的可比性。当前 ResearchAgent 对脚注的提取能力有限，建议对关键结论进行人工确认。

### 11.3 如何安全地测试交易策略

交易策略涉及真实资金，务必遵循以下安全流程：

1. **始终先在模拟盘运行**：所有交易所都支持模拟盘（Paper Trading）模式。在 `.env` 中将网络参数设置为模拟环境（如 OKX 的 `OKX_NETWORK=paper`），确保 `ALLOW_LIVE_TRADING` 保持为 `false`。
2. **设定明确的验证指标**：在开始测试前，确定你要观察的指标（收益率、最大回撤、夏普比率）和最低运行时长（建议至少覆盖一个完整的市场周期）。
3. **逐步放大仓位**：即使在实盘中，也应从最小仓位开始。确认策略在实盘环境中的表现与模拟盘一致后，再逐步增加仓位。
4. **保留退出机制**：确保所有策略都设置了止损参数，并且你能随时手动终止策略执行。

### 11.4 如何高效监控多只股票

当需要同时关注多只股票时，可以采用以下方法：

- **利用定时推送**：通过 NewsAgent 为不同的股票组合设置定时新闻推送任务，系统会在每个时间点自动汇总相关新闻。
- **批量研究请求**：通过编排器一次性提交包含多只股票的复合请求（如"比较 AAPL、MSFT、GOOGL 最新季度的营收增长"），系统会自动调度 ResearchAgent 处理。
- **文件监控**：为关注的股票分别设置 SEC 文件监控（ResearchAgent 的 `monitor_filings_and_notify` 技能），当有新的财报或重大事件文件提交时自动收到通知。

## 12. 常见问题

### Q1：可以同时和多个 Agent 对话吗？

不需要手动与多个 Agent 分别对话。ValueCell 的编排器（Orchestrator）负责多 Agent 协调。你只需在主界面提交请求，编排器会自动判断需要调度哪些 Agent，并将各 Agent 的结果整合后呈现给你。前端通过 `subagent_conversation` 组件清晰标记每个子 Agent 的贡献内容，让你能分辨不同部分的来源。

### Q2：ResearchAgent 支持哪些市场的财报？

当前 ResearchAgent 仅支持美国 SEC（证券交易委员会）的公开文件，包括 10-K 年报、10-Q 季报、13F 机构持仓报告和 8-K 重大事件报告。也就是说，它只能分析在美股上市的公司。A 股公司的财务研究请通过在线版本 valuecell.ai 使用。其他市场（港股、欧股等）的财报分析功能尚未实现。

### Q3：新闻数据来源是什么？

NewsAgent 通过网络搜索工具获取新闻数据，具体的数据来源会根据查询内容动态变化，并非固定使用某一家新闻服务。这意味着搜索结果的覆盖范围和相关性取决于查询关键词的精准度。建议在提问时使用具体的主题词和时间范围，以获得更聚焦的结果。

### Q4：如何知道系统正在使用哪个 Agent？

有两种方式可以观察当前活跃的 Agent：

1. **查看工具调用**：在前端的流式输出中，`tool_call_started` 和 `tool_call_completed` 事件会显示正在调用的工具名称和所属 Agent。
2. **查看子会话标记**：当多个 Agent 协作时，前端通过 `subagent_conversation` 组件标记每个 Agent 的会话边界，你可以清楚地看到 ResearchAgent 负责了哪部分分析、NewsAgent 贡献了哪些新闻内容。

如果你直接选择了某个 Agent 进行对话，则整个会话都由该 Agent 处理，不存在路由问题。

### Q5：交易策略的收益如何计算？

ValueCell 本身不内置策略收益指标的计算能力。策略执行结果（如成交记录、持仓状态）取决于所连接的交易所，相关指标（收益率、最大回撤、夏普比率等）需要通过以下方式获取：

- 利用 Agent 的研究分析功能间接获取分析结果
- 根据交易所返回的成交记录自行计算
- 参考交易所自带的策略分析工具

这一点在切换到实盘前尤其重要——确保你有能力独立评估策略的盈亏状况。
